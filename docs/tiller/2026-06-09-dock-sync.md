# Tiller: Dock-Sync — offline-first capture for the one place SailBook has no signal

*An overnight idea from Tiller. A draft, not a directive. Eric is the gate.*

---

## The idea

Every screen in SailBook assumes a network. That assumption is correct for
everyone except the one person whose data matters most: the instructor, who is
standing on a boat in the middle of the lake when the thing worth recording
actually happens.

**Dock-Sync** is a single offline-capable screen — the instructor's session
roster — that lets an instructor mark what happened *while it is happening*,
with zero connectivity, and reconcile automatically when the boat comes back
into wifi range at the dock. Marks land in a local IndexedDB outbox the instant
they're tapped (optimistic UI, a pattern already used elsewhere in the app), a
service worker keeps the page loadable with no signal, and a flush replays the
queue to the server the moment a connection returns.

Scope it to exactly one surface. Not an offline-first app — an offline-first
*roster*.

## Why it's worth it — and why *now*

Two things make this more than a convenience feature:

1. **It closes a capture gap that exists today.** Right now
   `instructor/sessions/[id]/page.tsx` is *read-only* — it displays attendance
   but the instructor can't mark it. Attendance is entered by the **admin**,
   later, at a desk, from whatever the instructor relayed. The witness and the
   recorder are different people at different times in different places. That's
   lossy even for attendance.

2. **It's a silent precondition for Phase 8.** Phase 8 (Skills & Tracking) is
   the marquee future arc: instructors mark individual skills demonstrated and
   executed, per student, per session — "tacking," "docking under power," "crew
   overboard." That's 8–12 checkboxes × 6 students = ~70 discrete observations
   per session. Reconstructing 70 observations *from memory at the dock that
   evening* isn't tracking; it's guessing. The entire value of granular skill
   records depends on capturing them **in the moment** — which is exactly the
   moment there is no signal.

   So the offline rail isn't a polish item bolted onto Phase 8. It's the thing
   that makes Phase 8 worth building at all. And the time to lay it is **before**
   Phase 8 hardcodes an online-only capture UI you'd have to tear out and
   retrofit. Build the rail, then run skills on top of it.

The leverage: one contained feature now turns a planned feature later from
"lossy" into "trustworthy," and fixes a present-day admin-relay friction on the
way past.

## Why you haven't already done this

This is the honest part, because it's where the idea was hiding.

SailBook is **online-by-architecture, all the way down**. Server Components
fetch directly via the Supabase server client; RLS gates every read; mutations
run through Server Actions. Offline-first reads, correctly, as a *foundational*
departure from all of that — a local store, a sync layer, conflict resolution —
i.e. a months-long rewrite and an adoption bet. Eric's own outside-tech radar
lists the local-first sync engines (Zero / ElectricSQL / PGlite) under the
marine use case and explicitly flags them **"evaluate, don't pick yet."** So the
whole area is parked behind a big, deferred architecture decision.

The reframe that unlocks it: **you don't need any of that here.** One instructor
owns one session's attendance rows. There is exactly one writer. There is no
concurrent edit to reconcile — so there is no CRDT, no sync engine, no Zero. The
load-bearing tech is a service worker + an IndexedDB outbox + a flush on
reconnect. And the existing write path is *already idempotent*:
`saveAttendance` upserts on conflict `(session_id, enrollment_id)`, so replaying
a queued write twice is safe by construction. The scary part of offline-first —
merge conflicts — simply doesn't exist at single-writer scope. That collapses a
deferred architecture bet into a contained, shippable feature.

## Build handoff

A Claude Code session on mill-dev can run this in one pass. Next.js 16 App
Router + Supabase + shadcn, per the existing stack.

### Approach

- **Don't adopt a sync engine.** Single-writer ⇒ last-write-wins per
  `(session_id, enrollment_id)` is the *entire* conflict model, and the server
  upsert already implements it. Pulling in Zero/Electric/TanStack DB here is
  decoration and a maintenance liability. (When true multi-writer offline shows
  up — it won't here — revisit.)
- **Three pieces:** (1) a narrowly-scoped service worker so the instructor
  session route loads with no signal; (2) an IndexedDB outbox that queues writes
  keyed by `(session_id, enrollment_id)` — the key gives free dedupe; (3) a thin
  POST route handler that replays a queued write through the same RLS-gated
  upsert. Server Actions need the framework runtime and aren't ideal as a replay
  target from a worker, so the outbox flushes to a plain route handler.
- **Optimistic by default:** tap → local state + outbox immediately; the network
  is a background reconcile, never a blocker. The app already uses optimistic
  updates (see `docs/QA.md`), so the UX pattern is familiar.
- **Make it installable** (web manifest) so instructors launch "the app" from
  the home screen in standalone mode, not a browser tab they'll lose.

### File-by-file

- `public/sw.js` — minimal hand-rolled service worker (no `next-pwa` dep —
  keeps control and avoids the weight). Scope to `/instructor/`. **Network-first
  for HTML** (so a new deploy is never masked by a stale shell), cache-first for
  static assets. Version the cache name off `NEXT_PUBLIC_APP_VERSION` and clean
  old caches on `activate`.
- `src/components/instructor/sw-register.tsx` — `'use client'`, registers the
  SW. Mount once in `src/app/(instructor)/layout.tsx`.
- `src/lib/offline/outbox.ts` — tiny IndexedDB wrapper (raw IDB or the ~1KB
  `idb`). Store `attendance_outbox`, keyPath `[session_id, enrollment_id]` so a
  re-tap overwrites the pending entry instead of stacking. API:
  `enqueue(record)`, `pending(sessionId)`, `remove(key)`, `flush()`.
  `flush()` POSTs each entry; on success removes it; on 401 keeps it (see
  gotchas); on network error leaves the queue intact.
- `src/app/api/instructor/attendance/route.ts` — `POST`. Authenticate via the
  Supabase server client (cookies); authorize with the **same DEC-007 check the
  page uses** (`course.instructor_id === user.id || session.instructor_id ===
  user.id`); upsert `session_attendance` on conflict `(session_id,
  enrollment_id)` — reuse the body of `saveAttendance`. Idempotent. Revalidate
  both the instructor and admin attendance paths.
- `src/components/instructor/session-capture.tsx` — `'use client'` island. Props:
  `sessionId`, `courseId`, SSR initial roster + attendance. Per student, a
  tap-to-cycle control (expected → attended → missed → excused). Each tap updates
  local state and `enqueue()`s. A sync-status chip: *"3 changes pending — will
  sync at the dock"* vs *"All synced."* Flush on `online` and on
  `visibilitychange`→visible; use the Background Sync API where present as a
  bonus, never as the primary path (see iOS gotcha).
- `src/app/(instructor)/instructor/sessions/[id]/page.tsx` — keep SSR initial
  load; replace the read-only attendance cells with `<SessionCapture>`. This is
  the screen turning from display-only into capture.
- `public/manifest.webmanifest` + `<link rel="manifest">` in the root layout —
  name, icons, `display: standalone`, scope `/instructor/`.
- `tests/instructor-offline.spec.ts` — Playwright with
  `context.setOffline(true)`: load online once, go offline, mark every student,
  assert pending count + that the route still renders offline, go online, assert
  the queue flushes and `session_attendance` matches. RLS is unchanged
  (`session_attendance` policies already cover this), so no new pgTAP — but add
  one case asserting the route rejects an instructor not assigned to the session.

### Gotchas / risks

- **iOS Safari has no Background Sync API**, and a PWA's service worker only runs
  while the page is open. Instructors are almost certainly on iPhones. So:
  flush on `online`/focus *while the tab is open*, and tell the instructor
  "keep it open until you see *All synced*." Do **not** promise true background
  sync on iOS — design for foreground flush and treat Background Sync as a
  progressive enhancement on Android/Chrome.
- **JWT expiry offline.** A long session can outlive the access token before the
  boat reconnects. On flush, a 401 must **not** drop queued writes — retain
  them, prompt re-login on reconnect, then flush. Losing an afternoon of marks to
  a silent 401 is the one unacceptable failure.
- **SW cache staleness.** Scope the worker to `/instructor/` only and keep HTML
  network-first so a fresh deploy isn't shadowed. Bump the cache version per
  release.
- **Stay in lane.** Offline queues *attendance status only* (and, later, skill
  checkoffs). Capacity, makeup creation, enrollment mutations, refunds — all stay
  server-side and online. Don't let the outbox grow to cover write paths whose
  conflict model isn't single-writer.
- **Don't let it sprawl into a rewrite.** The whole point is one screen. If a
  second surface wants offline, that's a follow-up, not this PR.

### Done when

- Open an instructor session online once. Switch to airplane mode (or DevTools
  → Offline). Mark every student. The chip shows the pending count. **Reload the
  page while still offline** — it loads (SW) with the marks intact (IDB).
  Reconnect; within a few seconds the outbox flushes and `session_attendance`
  rows match what was tapped; chip reads *All synced*.
- Playwright offline spec green; route-authorization case green.
- iPhone manual pass: same flow, tab kept open, marks land after reconnect.
