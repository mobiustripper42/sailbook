## Task 6: Apply account credit balance at checkout (#107)

**Completed:**
- Redemption half of the credit feature — students spend #106's balance at checkout. Full credit (balance ≥ price) skips Stripe entirely, no `payments` row (it's a Stripe transaction log; nothing Stripe-side happened) — the negative `credit_ledger` row is the audit trail. Partial credit reduces the Stripe line-item amount and carries the applied amount in session `metadata`; the webhook redeems it only after Stripe confirms the remainder was actually paid.
- `src/lib/enrollment-confirm.ts` (new) — extracted `confirmEnrollment()` (status/attendance/waitlist/notify) out of the webhook so the webhook and the new zero-charge checkout path share one implementation instead of drifting apart.

- `src/app/api/test/seed-credit` (new, dev-only) — seeds a credit balance for tests, with an `absolute` mode (offsets current balance to zero first, still append-only) so tests aren't at the mercy of leftover balance from other runs.
- `tests/checkout-credit.spec.ts` (new) — full-credit (no Stripe) and partial-credit (real Stripe test-mode checkout + hand-signed webhook) paths end-to-end, verifying the real session's metadata and reduced line-item amount, not just that the webhook trusts whatever payload it's handed.
- `npm run build` + lint + full pgTAP + targeted Playwright all green.

**Code review:** @code-review found three real issues, all fixed:
- **Ordering bug** — full-credit branch flipped enrollment to `confirmed` *before* the `credit_ledger` debit; a failed debit left a confirmed-but-uncharged enrollment. Fixed via `confirmWithFullCredit()`: debit first (fatal on failure — nothing was actually paid on this path), confirm second, mirroring the webhook's existing payment-before-status-flip order.
- **Double-spend race** — two concurrent `createCheckoutSession` calls for the same student on *different* courses could both read the same balance before either commits its debit (the `enrollments` unique constraint doesn't help across different courses). Closed at the DB level: new migration adds a `BEFORE INSERT` trigger on `credit_ledger` that serializes concurrent inserts per student (`pg_advisory_xact_lock`) and rejects any insert that would take the balance negative.
- **Webhook duplicate delivery** — a `payments` unique-violation (23505, the constraint that specifically guards duplicate concurrent webhook deliveries) was logged but not treated as a stop signal, so a genuine duplicate delivery could still double-redeem credit. Now short-circuits on that error code.

**Test flakiness (self-inflicted, found and fixed while verifying):** `checkout-credit.spec.ts`'s full-credit test was racing against **itself** — Playwright's desktop and mobile projects both run the whole file concurrently (`mode: 'serial'` only orders tests *within* one project's run), so both projects were redeeming the same `pw_student` balance at once. Fixed by scoping that test desktop-only too. Separately confirmed (not fixed, not fixable without a dedicated fixture) that this test file still races against `payment-e2e.spec.ts` when run together under 4 local workers — every Stripe-flow test in this repo shares `pw_student`, and there's no unused student fixture to dodge it with. Proven correct via isolated runs and `workers=1`; CI runs `workers: 1` so this specific flakiness never surfaces there — matches this project's already-documented "full suite pollution" pattern, not a new problem.
**Architect review (same PR, Eric's explicit request):** given the size of the credit feature ("kind of a big deal we just whipped up in a jiffy"), Eric asked for an independent @architect pass beyond @code-review — one-off override of DEC-S029's Fable-disabled default, specifically for this review.
